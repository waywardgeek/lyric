# lyric update - Legacy Go-source Design Doc

**Date**: 2026-06-04
**Author**: CodeRhapsody + Bill Cox

## Current Status

This document describes the Go-source understanding-file updater that existed
before Lyric became self-hosting. The active self-hosted `lyric` binary currently
supports `compile`, `test`, `fmt`, and `help`; it does not expose `verify`,
`update`, or `gen`.

The old implementation is preserved under `legacy/go-compiler/cmd/forge` and
still uses the Forge-era command/file names. Treat this document as design and
maintenance context for that legacy toolchain, or as a starting point for a
future self-hosted port.

## Problem

Understanding files have two maintenance problems:

1. **Creation friction**: writing an understanding file from scratch for an
   existing Go package requires transcribing exported types, fields, methods,
   and functions.
2. **Staleness**: as Go code evolves, the generated function signature index and
   structural declarations can drift.

The verifier catches drift but does not fix it. The updater/generator pipeline
was designed to generate and refresh understanding files from Go source.

## Three-Zone File Model

An understanding file has three zones separated by marker comments:

```lyric
lyric MyPackage {
  why: "..."

  doc "Architecture": """
    ...
  """

  struct Foo { ... }
  interface Bar { ... }

  source: ["foo.go", "bar.go"]
}

// --- index ---
//
// Auto-generated function/method index with signatures in Go syntax.
// DO NOT EDIT below this line - regenerated by update.
//
// file: foo.go
//
// NewFoo creates a Foo with default settings.
// func NewFoo(name string) *Foo    :23

// --- dependencies ---
//
// Auto-generated internal package dependencies.
//
// pkg/ast
// pkg/parser
```

### Zone 1 - Human-reviewed declarations

Zone 1 contains the `lyric { }` block: design rationale, docs, structural type
declarations, functions, relations, annotations, and `source:` directives.

The original design preserved Zone 1 byte-for-byte. The legacy implementation
evolved beyond that: it parses and re-emits Zone 1 so it can auto-add missing
exported declarations and, with pruning enabled, remove stale declarations.
Review Zone 1 diffs after running the updater.

### Zone 2 - Function/method index

Zone 2 is regenerated from Go source and grouped by the exact `source:` path. It
includes:

- the first Go doc line, when present
- the full Go function signature
- the source line number
- exported and unexported functions

### Zone 3 - Dependencies

Zone 3 lists internal Go package dependencies discovered from imports. Standard
library and third-party imports are excluded.

## Legacy Commands

In the preserved Go implementation, the relevant commands are under
`legacy/go-compiler/cmd/forge`:

```bash
forge verify <file.forge>
forge update [--prune] <file.forge>
forge gen <package-dir>
forge fmt <file.forge>
```

The names are historical. A self-hosted Lyric port should expose the equivalent
`.lyric` commands from the active `lyric` binary if these workflows are revived.

## Update Algorithm

1. Read the understanding file.
2. Parse Zone 1 and collect `source:` paths from all blocks.
3. Resolve source paths relative to the understanding file's directory.
4. Parse each Go source file with `go/parser` and `go/ast`.
5. Extract functions, methods, doc lines, signatures, line numbers, exported
   structs, interfaces, methods, functions, and aliases.
6. Update Zone 1 structural declarations from exported Go symbols.
7. Regenerate Zone 2 from the discovered functions.
8. Regenerate Zone 3 from internal package imports.
9. Write the re-emitted Zone 1 plus generated Zones 2 and 3.

## Generation Algorithm

For a new understanding file:

1. Scan the package directory for `.go` files, excluding `_test.go`.
2. Parse the files with `go/ast`.
3. Generate a Zone 1 block with placeholder `why: ""`, exported structural
   declarations, and a `source:` list.
4. Generate Zone 2 and Zone 3 using the same extraction code as update.
5. Write `<package-dir>/<package-name>.lyric` in a future Lyric port, or the
   corresponding legacy `.forge` file in the preserved implementation.

## Function Signature Extraction

The legacy tool uses Go's standard `go/parser` and `go/ast` packages:

- receiver from `FuncDecl.Recv`
- name from `FuncDecl.Name`
- params from `FuncDecl.Type.Params`
- returns from `FuncDecl.Type.Results`
- doc line from `FuncDecl.Doc`
- line number from `token.FileSet`

The index intentionally keeps signatures in Go syntax. It is a navigation aid,
not a Lyric type translation.

## Edge Cases

- Missing `source:`: fall back to scanning the understanding file's directory.
- Missing source file: warn and skip.
- No functions in a source file: still emit a `// file:` header.
- Multi-line docs: use the first line.
- Multiple blocks: collect and deduplicate all `source:` paths.
- Generated or build-tagged files: include them unless the source list excludes
  them.

## Non-Goals

- Semantic validation of narrative docs, invariants, relations, requires, or
  ensures clauses.
- Type-checking Go source.
- Translating Go function signatures into Lyric syntax for Zone 2.
- Cross-language source extraction without a language-specific extractor.

## Testing Strategy

- Unit tests for Go signature extraction.
- Unit tests for dependency filtering.
- Integration tests for update/gen/fmt command behavior.
- Regression tests for nested `source:` paths so function entries remain grouped
  under the same path that appears in the understanding file.
