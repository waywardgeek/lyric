# grok update ā€” Design Doc

**Date**: 2026-06-04
**Author**: CodeRhapsody + Bill Cox

## Problem

.grok files are the primary design artifact in GDD, but they have two maintenance problems:

1. **Creation friction**: Writing a .grok file from scratch for an existing Go package is tedious. You have to manually transcribe every exported type, field, method, and function.
2. **Staleness**: As code evolves, the function signature index drifts. Updating it requires manually re-reading Go source and editing the .grok file.

The verifier catches drift but doesn't fix it. We need a tool that **generates and updates** .grok files from Go source automatically.

## Design

### Three-Zone .grok File

A .grok file has three distinct zones, separated by marker comments:

```
// mypackage.grok ā€” the Grok package's own understanding

grok MyPackage {
  why: "..."

  doc "Architecture": """
    ...
  """

  // Structural declarations (structs, enums, interfaces, classes)
  struct Foo { ... }
  interface Bar { ... }

  source: ["foo.go", "bar.go"]
}

// --- index ---
//
// Auto-generated function/method index with signatures in Go syntax.
// DO NOT EDIT below this line ā€” regenerated by `grok update`.
//
// file: foo.go
//
// NewFoo creates a Foo with default settings.
func NewFoo(name string) *Foo                                         // :23
//
// Process runs the main processing loop.
func (f *Foo) Process(ctx context.Context) error                      // :45
//
// helper trims whitespace and normalizes.
func helper(s string) string                                          // :102
//
// file: bar.go
//
// Validate checks structural integrity.
func Validate(f *Foo) (bool, error)                                   // :15

// --- dependencies ---
//
// Auto-generated internal package dependencies.
// DO NOT EDIT below this line ā€” regenerated by `grok update`.
//
// pkg/ast
// pkg/parser
```

### Zone Definitions

**Zone 1 ā€” Human-curated (Grok syntax)**: Everything from the start of the file through the end of the last `grok { }` block. Contains `why:`, `doc` blocks, structural type declarations (structs, enums, interfaces, classes, functions, relations), annotations, and `source:` directives. This zone is **never modified** by `grok update`. The human (or AI with human review) maintains this.

**Zone 2 ā€” Function/method index (Go syntax)**: Starts at `// --- index ---` marker. Contains every function and method in the package, organized by source file, with:
- The first line of the Go doc comment (or `//` comment immediately preceding the func)
- The full Go function signature (receiver, name, params, returns)
- Line number as a trailing `// :NNN` comment
- Both exported and unexported functions included
- Blank line between entries for readability

This zone is **completely regenerated** on every `grok update` run.

**Zone 3 ā€” Dependencies (auto-generated)**: Starts at `// --- dependencies ---` marker. Lists internal (project-only) package imports, deduplicated across all source files. External stdlib and third-party imports are excluded. This zone is **completely regenerated** on every `grok update` run.

### Commands

#### `grok update <file.grok>`

Reads an existing .grok file, preserves Zone 1 verbatim, regenerates Zones 2 and 3 from the Go source files listed in `source:` annotations.

**Algorithm:**
1. Read the .grok file as raw text
2. Find the `// --- index ---` marker (or end of file if absent)
3. Preserve everything before the marker as Zone 1
4. Parse Zone 1 to extract `source:` file paths from grok blocks
5. Resolve source paths relative to the .grok file's directory
6. For each source file, use `go/parser` + `go/ast` to extract:
   - All function declarations (top-level and methods)
   - The first line of each function's doc comment
   - The exact source line number
   - The function signature as written in Go source
7. Organize by file, sort by line number within each file
8. For dependencies: collect all import paths across source files, filter to project-internal only (same module prefix from go.mod)
9. Write: Zone 1 + Zone 2 (index) + Zone 3 (dependencies)

**Key detail**: Zone 1 is preserved as raw text, not parsed and re-emitted. This means formatting, comments, and any non-standard content are kept exactly as-is. No `grok fmt` needed for this operation.

#### `grok gen <package-dir>`

Creates a new .grok file from scratch for a Go package.

**Algorithm:**
1. Find all `.go` files in the directory (excluding `_test.go`)
2. Parse all files with `go/ast`
3. Generate Zone 1:
   - `grok PackageName {` block
   - `why: ""` placeholder
   - For each exported type: generate Grok struct/interface declaration
   - `source: [list of .go files]`
   - `}`
4. Generate Zone 2: function index (same as `grok update`)
5. Generate Zone 3: dependencies (same as `grok update`)
6. Write to `<package-dir>/<package-name>.grok`

The generated Zone 1 has structural declarations but empty `why:` and no `doc` blocks ā€” the human fills those in. This is the scaffolding.

#### `grok fmt <file.grok>`

Reformats Zone 1 of a .grok file using standard formatting rules. Requires the parser to retain comments.

**Deferred** ā€” not part of this initial implementation. The parser needs comment-retention work first.

### Function Signature Extraction

We use `go/parser` and `go/ast` (the Go standard library) to extract signatures. This gives us:

- **Receiver**: from `FuncDecl.Recv` ā€” format as `(name Type)` or `(name *Type)`
- **Name**: from `FuncDecl.Name`
- **Params**: from `FuncDecl.Type.Params` ā€” reconstruct from Go AST field list
- **Returns**: from `FuncDecl.Type.Results` ā€” single value or tuple
- **Doc comment**: from `FuncDecl.Doc` ā€” take first line, trim `//` prefix
- **Line number**: from `FuncDecl.Pos()` via `token.FileSet`

We emit the signature as close to the original Go source as possible. No Grok translation ā€” this is a navigation index in the target language.

### Dependency Extraction

For each source file:
1. Walk `ast.File.Imports`
2. Extract import path strings
3. Read `go.mod` from the project root to get the module path
4. Filter: keep only imports that start with the module path
5. Strip the module path prefix to get relative package paths
6. Deduplicate across all source files
7. Sort alphabetically

### File Discovery

Source files come from two places:
- **`grok update`**: reads `source:` annotations from the existing .grok file
- **`grok gen`**: scans the directory for all `.go` files, excluding `*_test.go`

For `grok update`, if the `source:` annotation is missing or empty, fall back to scanning the .grok file's directory for `.go` files.

### Edge Cases

- **No source annotation**: Fall back to directory scan
- **Source file doesn't exist**: Warn and skip
- **No functions in a file**: Emit the `// file:` header with no entries
- **No doc comment**: Emit the signature with no comment line above it
- **Multi-line doc comments**: Take only the first line
- **Multiple grok blocks**: Collect `source:` from all blocks, deduplicate
- **Init functions**: Include `func init()` in the index
- **Generated files**: Include them (the human can add exclusion later if needed)
- **Build-tagged files**: Include all files regardless of build tags for completeness

## Implementation Plan

### Phase 1: `grok update` (MVP)

New file: `cmd/grok-update/main.go` (or add as subcommand to existing tools)

Actually ā€” better to unify under a single `grok` CLI with subcommands:

```
grok verify <file.grok>     (existing, move from grok-verify)
grok update <file.grok>     (new)
grok gen <package-dir>      (new, phase 2)
grok compile <file.gk>      (existing, move from grok-compile)
```

But for now, keep separate binaries to avoid disrupting existing tools:
- `cmd/grok-update/main.go` ā€” new

**Steps:**
1. Read .grok file, split at `// --- index ---` marker
2. Parse grok blocks (reuse existing parser) to extract `source:` paths
3. For each source file: parse with `go/parser`, extract functions
4. Format function signatures as Go text
5. Extract and filter dependencies
6. Write output: Zone 1 + generated Zone 2 + generated Zone 3

### Phase 2: `grok gen`

New file: `cmd/grok-gen/main.go`

**Steps:**
1. Scan directory for .go files
2. Parse all with `go/ast`
3. Generate Grok structural declarations from Go types
4. Generate function index + dependencies
5. Write complete .grok file

### Phase 3: Unified CLI

Merge all four commands under a single `grok` binary with subcommands.

### Phase 4: `grok fmt`

Extend the parser to retain comments, implement pretty-printer.

## Non-Goals

- **Modifying Zone 1**: `grok update` never touches the human-curated zone. If the human wants to add/remove structural declarations, they edit the .grok file manually.
- **Semantic analysis**: The function index is purely syntactic ā€” no type checking, no Grok type translation.
- **Cross-language support**: This design targets Go. Other languages (Python, Rust) would need their own extractors but the .grok file format stays the same.
- **IDE integration**: Future work. The function index with line numbers makes this possible.
- **Comment-aware parsing**: Deferred to Phase 4 (`grok fmt`).

## Testing Strategy

- Unit tests for signature extraction (various Go func patterns: methods, generics, variadic, multi-return)
- Unit tests for dependency filtering (module path matching)
- Integration test: run `grok update` on existing .grok files in this project, verify Zone 1 is preserved, Zone 2 matches expected output
- Golden file tests: known .go input ā†’ expected .grok output

## Open Questions

1. **Should the index include struct/interface type declarations too?** Currently only functions. Types are in Zone 1 (Grok syntax). But some types are missing from Zone 1 ā€” the completeness check would catch this, but having them in the index too would be redundant.

2. **Should unexported functions be included?** Yes for now ā€” they're useful for navigation. The `// file:` headers already group by file which provides organizational context. Could add a flag `--exported-only` later.

3. **Should we include function body line count?** E.g. `// :342 (45 lines)`. This would help estimate complexity at a glance. Low cost to add.
