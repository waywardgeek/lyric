# Runtime Module Design

## Executive Summary

The `runtime` module provides the foundational C runtime library required by the Forge compiler's C backend. It is implemented as a passive, header-only library that establishes the memory layout and operational semantics for Forge's high-level constructs in a C environment. By providing a suite of macros and inline functions, the runtime enables the C backend to transform complex Forge features—such as dynamic slices, length-prefixed strings, optional types, error results, tagged unions, and concurrency primitives—into efficient and readable C code. Its primary goal is to bridge the gap between Forge's expressive type system and C's low-level primitives while maintaining semantic equivalence and providing a robust execution environment.

## File Inventory

- [forge_runtime.h](forge_runtime.h): The core runtime header. It contains all macro definitions, type structures, and inline helper functions that constitute the Forge execution environment for C targets. It defines the fundamental data structures for slices, strings, optionals, and results, as well as the logic for concurrency and file I/O.

## Architecture and Data Flow

The `runtime` module is a passive component that does not maintain internal state or have an independent execution flow. Instead, it acts as a contract and a toolkit for the code generated by the [LIR module](../pkg/lir/design.md).

The architecture leverages C macros to implement a form of "template-based" type generation. Since C lacks native support for generics, the Forge C backend uses these macros to define concrete, specialized struct types for every unique instantiation of a generic Forge type. For example, the backend might generate `Slice_int32_t`, `Opt_string`, or `Chan_int32_t` using the provided macros. This approach ensures type safety at the C level and allows the compiler to generate code that is tailored to specific data types. Data flows through these structures as they are passed between functions, with the runtime providing the logic for common operations like resizing slices, wrapping values in optional containers, or managing thread-safe communication through channels.

## Interface Implementations

As a C header library, this module does not implement any Go interfaces. It serves as the target execution environment for the C code generator within the [LIR module](../pkg/lir/design.md).

## Public API

The Public API of the runtime consists of the macros and inline functions intended to be utilized by the Forge-generated C code. These are categorized by the Forge language features they support.

### Dynamic Slices
Forge slices are implemented as dynamic arrays with automatic resizing capabilities and support for sub-slicing. The `FORGE_SLICE_DEF` macro defines a struct containing a pointer to the data, the current length, and the total capacity. Slices can be initialized to an empty state using `forge_slice_empty`. The `forge_push` macro appends an element to a slice, performing a geometric resize if necessary, while `forge_pop` removes and returns the last element. Sub-slicing is supported via `forge_subslice`, which creates a new slice view sharing the underlying data. For convenience, `forge_slice_lit` allows creating a slice from a C initializer list, and `forge_contains` provides a linear search capability.

### Strings
Strings in Forge are implemented as length-prefixed byte slices, represented by the `forge_string` type (which is an alias for `ForgeSlice_uint8_t`). This design allows strings to contain embedded null bytes and ensures O(1) length operations. The `FORGE_STR` macro creates a `forge_string` from a C string literal at compile-time. For dynamic strings, `forge_str_from_cstr` and `forge_str_from_bytes` create heap-allocated copies, adding a hidden trailing null for C interop. The API includes functions for equality (`forge_str_eq`), lexicographic comparison (`forge_str_cmp`), and concatenation (`forge_str_concat`). Searching and manipulation are supported through `forge_str_contains`, `forge_str_index_of`, `forge_str_has_prefix`, `forge_str_has_suffix`, `forge_str_replace`, `forge_str_repeat`, `forge_str_join`, and `forge_str_trim`. Formatting is handled by `forge_sprintf`, which returns a heap-allocated string.

### Optional Types
Optional types are represented as a struct containing a boolean flag and the actual value. The `FORGE_OPT_DEF` macro defines these specialized structs. Values are wrapped using `forge_some` or represented as empty using `forge_none`. The `forge_isnull` helper checks for the absence of a value, and `forge_unwrap` extracts the value from the container.

### Error Results
Forge's idiomatic return pattern for operations that can fail is mapped to a result struct in C. `FORGE_RESULT_DEF` defines a struct containing a success value, an error message (as a `const char*`), and a boolean error flag. Results are constructed using `forge_ok` or `forge_err`, and the `forge_is_err` helper provides a quick check for failure.

### Concurrency Primitives
The runtime provides support for Forge's concurrency model using POSIX threads. `FORGE_CHAN_DEF` and `FORGE_CHAN_IMPL` are used to define and implement thread-safe, buffered or unbuffered channels. These channels support `make`, `send`, `recv`, `close`, and `free` operations. The `forge_spawn` function implements the Forge `spawn` keyword by launching a function in a new detached thread.

### Utility Helpers
The runtime includes various utilities for common tasks. File I/O is supported by `forge_read_file` and `forge_write_file`, while `forge_getwd` provides the current working directory. Path manipulation is handled by `forge_path_dir`, `forge_path_base`, and `forge_path_ext`. Conversion functions include `forge_itoa`, `forge_atoi`, `forge_parse_float`, and `forge_char_to_string`. Case conversion is provided by `forge_toupper` and `forge_tolower`. The `forge_hash_string` function implements the FNV-1a hashing algorithm for strings.

### Tagged Unions
For ad-hoc union types, the runtime provides the `ForgeUnion` struct, which uses an integer tag to identify the active member. Specialized constructor functions like `forge_union_i32` and `forge_union_string` initialize these unions with the correct tag and value.

## Implementation Details

### Memory Management
The runtime currently utilizes the standard C library's `malloc`, `realloc`, and `free` for memory management. A critical implementation detail is that the runtime does not yet include a garbage collector. Consequently, heap-allocated objects—including slices, strings created by helper functions, and class instances—are currently leaked during execution. This is a known limitation that is acceptable for short-lived compiler tasks but will require a GC (such as Boehm-Demers-Weiser) for broader use.

### Slice Growth Strategy
The `forge_push` macro implements a geometric growth strategy to ensure amortized O(1) append operations. When a slice's length reaches its capacity, the capacity is doubled. If the slice was previously empty, it is initialized with a default capacity of 4. This minimizes the number of reallocations required as a slice grows.

### String Handling
Strings in the Forge C runtime are represented by the `forge_string` struct, which contains a pointer to the data and its length. This design allows for O(1) length retrieval and supports strings with embedded null bytes. For compatibility with standard C functions that expect null-terminated strings, heap-allocated `forge_string` objects carry a hidden trailing `\0` byte immediately following the data. This trailing byte is not included in the `.len` field.

### Concurrency Implementation
Channels are implemented using a circular buffer protected by a `pthread_mutex_t`. Two condition variables (`not_empty` and `not_full`) are used to manage blocking send and receive operations. `forge_spawn` uses `pthread_create` and `pthread_detach` to provide fire-and-forget thread execution.

### Tagged Union Dispatch
The `ForgeUnion` type uses an integer tag to identify the active member of its internal union. This allows the generated C code to perform safe type dispatching (e.g., in `match` statements) by checking the tag before accessing the union's data.

## Dependencies

- **Standard C Library**: The runtime depends on standard headers including `<stdio.h>`, `<stdlib.h>`, `<stdint.h>`, `<stdbool.h>`, `<string.h>`, `<stdarg.h>`, `<ctype.h>`, and `<unistd.h>`.
- **POSIX Threads**: The concurrency primitives require the `pthread` library.
- **LIR Module**: This module is a critical dependency for the C backend in [pkg/lir](../pkg/lir/design.md), which relies on the macros and types defined here to emit valid C code.

## Technical Debt and Future Work

- **Garbage Collection**: Integrating a garbage collector is the highest priority to prevent memory leaks in long-running Forge programs compiled to C.
- **Map Implementation**: A robust hash map implementation is required to support Forge's `map<K, V>` type in the C backend.
- **Error Handling Refinement**: The current error result uses a simple string for error messages. This should be expanded to support more complex, structured error types as the language evolves.
- **Thread Safety**: While channels are thread-safe, other runtime primitives (like slices) are not and will require synchronization if shared across threads.
